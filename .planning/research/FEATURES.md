# 功能特性分析

**领域:** 低代码风控规则引擎
**调研时间:** 2025-03-26
**整体置信度:** HIGH

## 行业标配（Table Stakes）

用户期望风控规则引擎具备的功能。缺失这些功能会让产品感觉不完整或不可用。

| 功能 | 预期原因 | 复杂度 | 备注 |
|------|---------|--------|------|
| **规则可视化配置** | 非技术人员需要独立配置规则，避免依赖开发 | 中 | 支持拖拽式流程编排，降低业务人员使用门槛 |
| **规则表单配置** | 简单规则（字段比较、数值范围）需要快速配置入口 | 低 | 通过表单快速配置单条件规则，提升效率 |
| **同步 API 执行** | 支付链路需要实时决策，阻塞等待结果 | 中 | 需要优化到 <50ms 响应时间 |
| **规则版本管理** | 规则变更风险高，需要追溯历史版本 | 中 | 支持版本创建、修改、删除、回滚 |
| **规则一键回滚** | 新规则出现问题需快速恢复，避免业务损失 | 中 | 与版本管理配合，支持秒级切换 |
| **规则灰度发布** | 降低新规则上线风险，按比例放量验证 | 高 | 按百分比流量切换，支持 A/B 测试 |
| **规则执行监控** | 需要了解规则命中情况、执行效果 | 中 | 记录规则命中统计、执行日志 |
| **规则测试验证** | 规则上线前需验证逻辑正确性 | 中 | 支持模拟数据测试、实时调试 |
| **特征获取多策略** | 优先使用入参特征，降级到外部特征平台 | 高 | 减少外部依赖，提升性能，保持灵活性 |
| **审计日志** | 金融合规要求，需要记录所有规则变更和执行 | 中 | 记录谁在何时做了什么操作 |
| **高性能执行** | 支付链路要求毫秒级响应，不能引入明显延迟 | 高 | 行业标准：<100ms，优秀：<50ms |
| **规则持久化** | 规则需要持久化存储，避免丢失 | 低 | 使用 PostgreSQL + JPA 持久化规则元数据 |
| **规则 DSL 存储** | 规则以 DSL 形式存储，便于动态加载和执行 | 中 | Groovy DSL 动态加载，JVM 生态兼容 |

**关键发现：**
- **性能是硬性要求**：行业共识是反欺诈决策必须在 **100ms 以内**完成，优秀系统优化到 **50-75ms**【来源：多个反欺诈系统设计文章】
- **版本管理不可妥协**：风控规则变更风险极高，灰度发布和快速回滚是必备能力【来源：美团 Zeus、有赞风控实践】
- **低代码是趋势**：可视化配置和表单配置降低业务人员使用门槛，是主流方向【来源：DecisionRules、GoRules 等产品】

## 差异化功能（Differentiators）

让产品在竞争中脱颖而出的功能。用户不一定预期，但会高度认可。

| 功能 | 价值主张 | 复杂度 | 备注 |
|------|---------|--------|------|
| **同步/异步混合执行模式** | 同步满足实时决策需求，异步满足非实时场景，一套引擎两种模式 | 高 | 这是项目的核心差异化能力 |
| **决策表支持** | 对于多维度条件组合，决策表比流程图更直观 | 中 | 适合复杂的评分卡、费率计算场景 |
| **规则模板库** | 预置常见反欺诈规则模板，用户开箱即用 | 中 | 降低新用户上手门槛，如"同一IP多次注册"、"短时间内大额交易" |
| **规则依赖分析** | 分析规则之间的依赖关系，避免冲突和死循环 | 高 | 帮助用户理解规则影响范围，降低配置错误风险 |
| **规则效果分析** | 统计规则命中率、误判率、拦截率等指标 | 中 | 帮助业务人员优化规则策略 |
| **特征自动推荐** | 根据配置的规则条件，推荐相关特征字段 | 低 | 减少用户查找特征的时间 |
| **规则冲突检测** | 自动检测规则之间的逻辑冲突（如矛盾的阈值） | 高 | 提前发现配置错误，避免线上问题 |
| **规则智能排序** | 根据命中率和执行时间自动优化规则执行顺序 | 中 | 提升整体执行性能 |
| **多环境支持** | 支持开发、测试、生产环境规则隔离 | 中 | 企业级客户的必备需求 |
| **规则导入导出** | 支持批量导入导出规则，便于跨系统迁移 | 低 | 便于规则复用和备份 |

**关键发现：**
- **同步/异步混合模式**是核心差异点：大多数规则引擎只支持同步或异步一种模式，混合模式可以覆盖更广泛的业务场景
- **决策表**是高价值功能：对于风控评分、费率计算等场景，决策表比流程图更直观、更易维护
- **规则分析和优化工具**是高级功能：帮助用户理解规则效果、优化性能，提升产品专业度

## 反模式（Anti-Features）

明确不构建的功能。避免过度设计，保持产品聚焦。

| 反模式 | 避免原因 | 替代方案 |
|--------|---------|---------|
| **工作流引擎功能** | 规则引擎不是用来处理工作流或流程执行的【来源：腾讯云开发者社区】 | 使用专门的 BPM 工具或工作流引擎 |
| **复杂计算逻辑** | 规则引擎应该专注于条件判断，不应该嵌入复杂计算 | 预留扩展点，计算在外部系统完成 |
| **机器学习模型集成** | v1 版本不做，增加复杂度和学习成本 | 后续版本考虑，通过外部服务集成 |
| **实时流处理** | v1 仅支持同步 API + 异步事件，不做流式计算 | 后续版本考虑 Kafka 流处理 |
| **移动端管理界面** | v1 仅 Web 端，移动端配置体验不佳 | 后续版本考虑响应式设计 |
| **AI 辅助规则生成** | 技术不成熟，可能生成错误规则 | 后续版本考虑，提供规则推荐 |
| **规则自然语言编辑** | 技术不成熟，容易产生歧义 | 使用可视化配置和 DSL |
| **分布式规则执行** | 增加系统复杂度，单机性能足够 | 单机多线程执行，预留分布式扩展点 |

**关键发现：**
- **工作流 ≠ 规则引擎**：这是常见的混淆，工作流引擎处理流程编排，规则引擎处理条件判断，应该各司其职【来源：多个技术文章】
- **避免过度设计**：AI、ML、流处理等功能在 v1 版本不做，聚焦核心价值
- **预留扩展点**：虽然不实现复杂计算和 ML 模型，但要预留扩展点，便于后续集成

## 功能依赖关系

```
规则版本管理 → 规则灰度发布（灰度需要多版本共存）
规则版本管理 → 规则一键回滚（回滚需要历史版本）
规则可视化配置 → 规则测试验证（配置后需要验证）
规则同步执行 → 特征获取多策略（同步执行需要高性能特征获取）
规则执行监控 → 规则效果分析（效果分析基于执行监控数据）
规则模板库 → 规则可视化配置（模板可快速生成可视化规则）
```

## MVP 建议

**优先级 1（必须）：**
1. 规则可视化配置（拖拽式流程编排）
2. 规则表单配置（简单条件快速配置）
3. 同步 API 执行（<50ms 响应）
4. 规则版本管理（创建、修改、删除）
5. 规则一键回滚（秒级切换）
6. 规则测试验证（模拟数据测试）
7. 特征获取多策略（入参优先，降级外部）
8. 规则持久化（PostgreSQL + JPA）
9. 规则 DSL 存储（Groovy）

**优先级 2（重要）：**
1. 规则灰度发布（按比例流量切换）
2. 规则执行监控（命中统计、执行日志）
3. 审计日志（操作记录）
4. 决策表支持（多维度条件组合）
5. 异步事件执行（非实时场景）

**优先级 3（增强）：**
1. 规则效果分析（命中率、误判率）
2. 规则模板库（预置常见规则）
3. 规则冲突检测（逻辑冲突检测）
4. 规则依赖分析（依赖关系分析）
5. 多环境支持（开发、测试、生产隔离）

**延后（后续版本）：**
- AI 辅助规则生成（技术不成熟）
- 机器学习模型集成（v2 版本）
- 实时流处理（v2 版本）
- 移动端管理界面（v2 版本）

## 复杂度分析

| 功能领域 | 复杂度 | 关键挑战 |
|---------|--------|---------|
| 规则配置 UI | 高 | 需要支持拖拽、连线、条件嵌套，交互复杂 |
| 规则执行引擎 | 高 | 需要高性能、动态加载、同步异步混合 |
| 版本管理 | 中 | 多版本共存、灰度切换、数据一致性 |
| 特征获取 | 中 | 多策略降级、缓存优化、性能保证 |
| 监控统计 | 中 | 大量数据实时统计、查询优化 |
| 测试验证 | 中 | 模拟数据生成、执行结果展示 |
| 审计日志 | 低 | 标准日志记录和查询 |

**关键挑战：**
1. **性能优化**：在 <50ms 内完成规则执行是最大挑战，需要优化特征获取、规则匹配、结果返回整个链路
2. **灰度发布**：多版本规则共存、按比例流量切换、数据一致性是技术难点
3. **可视化 UI**：拖拽式流程编排需要精心设计交互，避免过于复杂
