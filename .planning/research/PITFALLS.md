# Domain Pitfalls Research

**Domain:** 风控规则引擎（Risk Control Rule Engine）
**Researched:** 2025-03-26
**Confidence:** HIGH

## Critical Pitfalls

### Pitfall 1: Groovy 动态类加载导致的内存泄漏

**What goes wrong:**
每次执行 Groovy 脚本时动态创建类，这些类被加入到 Groovy 管理的缓存内，GC 时无法释放，最终耗尽 Metaspace 空间，导致 OOM。类加载过程中的同步机制还会导致多线程阻塞。

**Why it happens:**
- GroovyClassLoader 和 GroovyShell 在每次编译脚本时都会创建新的类加载器
- 动态类被缓存但从不释放，导致 Metaspace/PermGen 持续增长
- 频繁的脚本重新编译会增加 JVM 内存中的 CodeCache
- 类加载存在同步机制，多线程环境下造成大量线程阻塞

**How to avoid:**
- 使用脚本缓存机制，避免重复编译相同脚本
- 对 GroovyClassLoader 使用 `removeClassCache()` 或定期清理缓存
- 使用单例 ClassLoader 而非每次创建新的
- 设置 Metaspace 大小监控告警（如使用率超过 80% 告警）
- 实现脚本版本管理，相同版本的脚本复用已编译的 Class
- 使用 Groovy 的 `parseClass()` 而非 `evaluate()` 以获得更好的控制

**Warning signs:**
- Metaspace 使用率持续增长（即使应用空闲时）
- 频繁的 Full GC 但内存回收效果不佳
- `jmap -histo` 显示大量 Groovy 动态类
- 应用响应时间逐渐变慢，特别是规则执行操作
- 线程池阻塞，大量线程等待类加载

**Phase to address:**
Phase 1（核心规则引擎）— 必须在第一个迭代中建立脚本缓存和类加载管理机制，这是基础架构问题，后期修复成本极高。

### Pitfall 2: 缺乏沙箱隔离导致代码注入攻击

**What goes wrong:**
攻击者利用 Groovy 的动态特性绕过沙箱限制，通过反射机制（`getClass().forName()`）、闭包委托、MetaClass 篡改、AST 转换等方式执行任意代码，导致数据泄露、系统控制权丧失。

**Why it happens:**
- Groovy 是一门动态语言，提供了强大的反射和元编程能力
- 默认的 GroovyShell 没有任何安全限制
- 简单的黑名单式沙箱容易被绕过（如 ElasticSearch Groovy 沙箱绕过漏洞 CVE-2015-1427）
- 业务人员配置规则时可能无意中引入危险操作

**How to avoid:**
- 使用 `CompilerConfiguration` 自定义 Groovy 安全管理器
- 禁用危险类：`System.class`、`Runtime.class`、`ProcessBuilder.class`、`File.class` 等
- 使用 SecurityManager 白名单机制，只允许特定类和方法
- 实现自定义 Import 范围控制，限制可访问的包
- 脚本执行超时控制（使用 Resilience4j 或类似机制）
- 在独立的 ClassLoader 中执行脚本，限制其父类加载器访问
- 考虑使用 GraalVM JavaScript 等更安全的替代方案

**Warning signs:**
- 规则中包含 `getClass()`、`forName()`、`invoke()` 等反射调用
- 规则尝试访问文件系统、网络、进程等系统资源
- CPU 使用率异常高（可能是死循环或挖矿脚本）
- 安全扫描发现可疑的脚本模式
- 日志中出现权限拒绝异常但仍能继续执行

**Phase to address:**
Phase 1（核心规则引擎）— 安全机制必须在第一期实现，任何安全漏洞在风控系统都是致命的。

### Pitfall 3: 规则执行超时导致整个服务雪崩

**What goes wrong:**
单个规则的恶意或错误配置（如死循环、大量嵌套循环、复杂的笛卡尔积计算）导致执行超时，耗尽线程池，导致整个风控服务不可用，影响支付链路。

**Why it happens:**
- 业务人员配置规则时缺乏性能意识
- Groovy 脚本可以实现任意复杂的逻辑，包括无限循环
- 没有执行时间限制和熔断机制
- 线程池配置不当，单个慢请求耗尽所有资源

**How to avoid:**
- 为每个规则执行设置超时时间（建议 50-100ms，不超过项目要求的 50ms 目标）
- 使用 Resilience4j 或 Hystrix 实现熔断机制
- 实现规则复杂度评估，拒绝或警告高风险配置
- 使用独立线程池执行规则，隔离风险
- 实现规则执行前的静态分析，检测可能的死循环
- 监控 P99、P95 响应时间，设置告警阈值
- 为不同优先级的规则配置不同的超时时间

**Warning signs:**
- 规则执行 P99 时间超过 50ms
- 线程池队列持续积压
- 规则执行超时告警频繁触发
- 某个规则执行时间明显异常（如 >1s）
- 服务整体响应时间变慢

**Phase to address:**
Phase 1（核心规则引擎）— 超时和熔断机制是核心稳定性保障，必须在第一个迭代实现。

### Pitfall 4: 特征获取性能瓶颈导致无法满足 50ms 延迟要求

**What goes wrong:**
规则执行过程中需要获取大量特征（用户画像、历史行为、设备信息等），如果特征获取逻辑设计不当，会导致大量的外部 RPC 调用或数据库查询，最终无法满足 50ms 的延迟要求。

**Why it happens:**
- 过度依赖外部特征平台，每个规则触发多次 RPC 调用
- 没有实现特征缓存，重复获取相同特征
- 特征获取逻辑分散在规则中，无法批量优化
- 没有实现特征预加载或异步加载
- 数据库查询未优化，N+1 查询问题

**How to avoid:**
- 优先使用入参携带的特征，减少外部依赖
- 实现特征缓存机制（本地缓存 + 分布式缓存）
- 支持特征批量获取，减少网络往返次数
- 实现特征预加载，在规则执行前一次性获取所有所需特征
- 监控特征获取耗时，识别慢查询
- 对高频特征使用本地缓存（如 Caffeine）
- 实现特征降级策略，超时时返回默认值

**Warning signs:**
- 规则执行总时间中，特征获取占比超过 50%
- 单个特征获取耗时超过 10ms
- 规则执行火焰图显示大量等待时间
- 外部服务调用 QPS 异常高
- 数据库慢查询告警频繁

**Phase to address:**
Phase 1（核心规则引擎）— 特征获取是性能的关键，必须在第一期设计和实现优化策略。

### Pitfall 5: 规则版本冲突导致数据不一致

**What goes wrong:**
多个业务人员同时修改同一规则，或者灰度发布期间新旧版本同时存在，导致规则执行结果不一致，无法确定哪个版本是正确的，影响风控决策的准确性。

**Why it happens:**
- 缺乏乐观锁或版本控制机制
- 灰度发布期间流量路由逻辑错误
- 规则缓存未正确失效
- 并发编辑冲突检测缺失
- 没有实现规则的原子性更新

**How to avoid:**
- 使用乐观锁（JPA `@Version` 注解）防止并发修改冲突
- 实现规则编辑锁定机制，同一时间只能一人编辑
- 灰度发布期间使用明确的版本标识，避免混淆
- 实现规则缓存的版本感知失效机制
- 记录规则变更审计日志，可追溯每次修改
- 灰度验证通过后才切换流量，避免直接覆盖
- 实现规则发布的审批流程

**Warning signs:**
- 规则执行结果与配置不符
- 业务人员抱怨"规则没生效"
- 相同请求在不同时间返回不同结果（非灰度场景）
- 规则版本号频繁回退
- 审计日志显示同一时间有多次修改

**Phase to address:**
Phase 2（规则管理与版本控制）— 版本管理是核心需求，必须在第二阶段完整实现。

---

### Pitfall 6: 灰度验证不充分导致生产事故

**What goes wrong:**
新规则上线前未充分验证，直接全量发布，导致误伤正常用户、误拒正常交易，造成业务损失和用户投诉。

**Why it happens:**
- 缺乏灰度发布机制，直接全量上线
- 灰度流量比例过大或过小，无法有效验证
- 灰度期间未充分监控关键指标
- 没有实现快速回滚机制
- 业务人员急于上线，跳过验证步骤
- 灰度验证数据不足，无法得出统计显著性结论

**How to avoid:**
- 实现灰度发布机制（标记、双跑、回溯三种状态）
- 灰度规则实时执行但不返回决策（标记状态）
- 灰度规则使用历史数据回放验证（回溯状态）
- 灰度流量从小到大逐步增加（1% -> 5% -> 10% -> 50% -> 100%）
- 监控灰度规则的关键指标：命中率、误伤率、执行时间
- 实现一键回滚到上一个稳定版本
- 强制验证策略：修改生产规则必须先灰度验证
- 灰度期间设置告警阈值，异常时自动回滚

**Warning signs:**
- 新规则上线后客诉激增
- 交易拦截率异常变化
- 业务人员报告"正常用户被拒"
- 灰度规则命中率过低（可能配置错误）
- 回溯验证结果与预期不符

**Phase to address:**
Phase 2（规则管理与版本控制）— 灰度发布是风控系统的核心安全机制，必须在第二阶段实现。

### Pitfall 7: 规则复杂度失控导致维护困难

**What goes wrong:**
随着业务发展，规则数量和复杂度快速增长，出现大量重复规则、冲突规则、冗余规则，导致系统难以维护，修改一个规则可能影响其他规则，排查问题困难。

**Why it happens:**
- 缺乏规则复用机制，每个场景都创建新规则
- 没有规则组管理，规则散乱无序
- 业务人员不理解规则之间的依赖关系
- 缺乏规则审查和重构机制
- 规则命名不规范，难以理解其用途
- 没有实现规则的分层和模块化

**How to avoid:**
- 实现规则组（Rule Group）机制，支持规则聚类管理
- 提供扩展函数、累计因子、决策表因子等封装逻辑
- 支持规则的复用和差异化应用
- 实现规则依赖分析和影响范围评估
- 定期进行规则审查，识别重复和冗余规则
- 提供规则命名规范和最佳实践文档
- 实现规则的可视化展示，帮助理解规则关系
- 建立规则评审机制，重要规则变更需要评审

**Warning signs:**
- 规则数量超过 1000 个且无组织
- 业务人员抱怨"找不到规则"
- 修改一个规则导致多个场景受影响
- 存在大量相似但略有不同的规则
- 规则执行时间异常长（可能是规则冲突导致的重复执行）

**Phase to address:**
Phase 2（规则管理与版本控制）— 规则组织和管理是可维护性的关键，必须在第二阶段建立规范。

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| 硬编码规则逻辑 | 快速实现第一个版本 | 业务无法独立配置，每次修改需要开发 | MVP 验证阶段，但必须在 Phase 2 替换 |
| 跳过规则复杂度检查 | 快速上线复杂规则 | 性能问题难以排查，系统稳定性下降 | 绝不 acceptable，必须在 Phase 1 实现基础检查 |
| 同步获取所有特征 | 实现简单 | 性能无法优化，无法满足 50ms 要求 | 仅在 MVP 验证阶段可接受，Phase 1 必须优化 |
| 不实现沙箱隔离 | 减少开发工作量 | 安全漏洞风险高，可能被攻击 | 绝不 acceptable，必须在 Phase 1 实现 |
| 规则直接存储为字符串 | 存储简单，实现快 | 难以实现版本管理、差异对比、灰度发布 | MVP 阶段可接受，Phase 2 必须优化 |
| 不实现监控告警 | 减少初期工作量 | 问题发现晚，故障影响时间长 | 仅在 MVP 阶段可接受，Phase 1 必须实现基础监控 |
| 使用数据库悲观锁 | 实现简单 | 高并发场景性能差 | 低并发场景可接受，高并发必须用乐观锁 |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| 外部特征平台 | 每个规则单独调用特征接口，导致 N 次网络调用 | 批量获取特征，使用缓存，优先使用入参携带的特征 |
| PostgreSQL 规则存储 | 直接存储 Groovy 脚本字符串，难以版本管理 | 使用 JPA 实体管理规则元数据，版本号 + 乐观锁 |
| Groovy 脚本执行 | 每次创建新的 GroovyShell，导致内存泄漏 | 复用 GroovyShell，实现脚本缓存，定期清理类加载器 |
| Redis 特征缓存 | 缓存无过期时间，导致数据陈旧 | 设置合理的过期时间，实现缓存预热和更新机制 |
| 消息队列异步事件 | 消息无重试机制，丢失导致风控遗漏 | 实现消息持久化、重试、死信队列机制 |
| 监控告警系统 | 只监控规则执行成功率，忽略耗时和异常 | 监控 P50/P95/P99 耗时、规则命中率、特征获取耗时、异常率 |
| 前端可视化配置 | 直接暴露 Groovy 语法，业务人员难以理解 | 提供表单配置和可视化流程，底层生成 Groovy 代码 |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| 规则执行超时 | P99 耗时 >50ms，线程池耗尽 | 实现超时熔断，规则复杂度检查，执行前预分析 | 单个规则执行时间 >100ms 时 |
| 特征获取 N+1 问题 | 规则执行耗时大部分在特征获取上 | 批量获取特征，实现特征缓存，预加载 | 特征数量 >10 个时 |
| Groovy 脚本重复编译 | CPU 使用率高，Metaspace 增长 | 实现脚本缓存，复用已编译的 Class | 规则变更频率 >10 次/天时 |
| 规则笛卡尔积计算 | 内存溢出，执行时间指数增长 | 规则静态分析，禁止危险模式 | 规则涉及多个 List 的 from 语句时 |
| 规则缓存失效导致全量重加载 | 数据库压力大，规则更新慢 | 实现增量更新，版本感知缓存失效 | 规则数量 >1000 时 |
| 灰度规则全量执行 | 系统负载翻倍，性能下降 | 灰度规则采样执行，或异步执行 | 灰度规则比例 >50% 时 |
| 日志打印过于详细 | 磁盘 I/O 高，规则执行变慢 | 实现日志级别控制，异步日志，采样日志 | 高并发场景（>1000 QPS）时 |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| 不限制 Groovy 脚本的类访问 | 攻击者可执行任意代码，窃取数据或破坏系统 | 实现沙箱隔离，使用 SecurityManager，禁用危险类 |
| 规则配置权限控制不当 | 业务人员可能修改不该修改的规则，或查看敏感规则 | 实现基于角色的权限控制，规则分级管理 |
| 规则审计日志不完整 | 无法追踪谁在什么时间修改了什么规则，事故难以回溯 | 记录所有规则变更的审计日志，包括操作人、时间、变更内容 |
| 灰度规则结果泄露 | 攻击者通过灰度规则探测系统风控策略 | 灰度规则结果不返回给上游，仅用于内部验证 |
| 特征数据越权访问 | 规则可能访问不该访问的特征，导致隐私泄露 | 实现特征访问控制，规则只能访问授权的特征 |
| 规则导入导出缺乏校验 | 导入恶意规则导致系统被攻击 | 实现规则导入的格式校验和安全扫描 |
| API 缺乏认证和限流 | 攻击者通过 API 进行暴力破解或拒绝服务攻击 | 实现 API 认证、限流、签名验证 |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| 可视化配置过于复杂 | 业务人员学习成本高，不愿使用，仍然依赖开发 | 提供表单配置（简单规则）和可视化配置（复杂规则）两种方式 |
| 规则配置缺乏即时验证 | 业务人员配置后不知道是否正确，上线后发现错误 | 实现配置时的实时语法检查和逻辑验证 |
| 规则执行结果难以理解 | 业务人员不知道规则为什么命中或未命中 | 提供规则执行回放功能，展示每条规则的执行过程和结果 |
| 规则查找困难 | 业务人员找不到想要的规则，重复创建 | 实现规则的分类、标签、搜索、收藏功能 |
| 规则修改影响范围不明确 | 业务人员不敢修改规则，担心影响其他场景 | 实现规则依赖分析和影响范围评估，修改前显示受影响场景 |
| 灰度验证结果不直观 | 业务人员无法判断灰度规则效果如何，不敢全量上线 | 提供灰度数据的可视化对比，命中率和误伤率的趋势图 |
| 移动端不支持 | 业务人员出差时无法紧急修改规则 | Phase 2 之后考虑支持移动端审批和紧急修改 |

## "Looks Done But Isn't" Checklist

- [ ] **规则版本管理**: 往往缺少版本回滚能力 — verify 能一键回滚到任意历史版本，且回滚后规则立即生效
- [ ] **灰度发布机制**: 往往缺少灰度流量控制 — verify 支持按比例（1%/5%/10%/50%）设置灰度流量，且可以实时调整
- [ ] **规则执行监控**: 往往只监控成功率 — verify 同时监控 P50/P95/P99 耗时、规则命中率、特征获取耗时
- [ ] **规则冲突检测**: 往往无法检测规则之间的冲突 — verify 实现规则依赖分析，检测冲突规则和重复规则
- [ ] **规则审计日志**: 往往只记录变更，不记录原因 — verify 记录规则变更的原因、审批人、关联需求
- [ ] **规则测试环境**: 往往缺少独立的测试环境 — verify 支持规则在测试环境验证后再发布到生产
- [ ] **规则导出导入**: 往往不支持批量导出和导入 — verify 支持规则批量导出、导入、格式校验
- [ ] **规则权限控制**: 往往只有简单的角色控制 — verify 实现基于场景的规则权限，不同业务只能管理自己的规则
- [ ] **规则性能分析**: 往往无法分析慢规则 — verify 实现规则执行火焰图，识别性能瓶颈
- [ ] **特征管理**: 往往缺少特征的定义和文档 — verify 实现特征目录，包含特征的定义、类型、数据来源

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Groovy 内存泄漏 | HIGH | 1. 立即清理规则缓存 2. 重启应用释放 Metaspace 3. 实现脚本缓存和类加载管理 4. 添加 Metaspace 监控告警 |
| 规则执行超时 | MEDIUM | 1. 立即下线问题规则 2. 检查规则逻辑，识别性能问题 3. 优化规则或拆分成多个简单规则 4. 实现规则执行超时熔断 |
| 沙箱绕过漏洞 | CRITICAL | 1. 立即停止规则引擎服务 2. 审计所有规则，识别恶意规则 3. 修复沙箱机制，加强安全限制 4. 实现规则安全扫描 5. 逐步恢复规则执行 |
| 灰度验证不充分 | HIGH | 1. 立即回滚到上一个稳定版本 2. 分析客诉和拦截率数据 3. 优化规则逻辑 4. 重新进行灰度验证 5. 实现灰度自动回滚机制 |
| 特征获取性能瓶颈 | MEDIUM | 1. 识别慢特征，优化查询或增加缓存 2. 实现特征批量获取 3. 添加特征获取耗时监控 4. 实现特征降级策略 |
| 规则版本冲突 | LOW | 1. 回滚到正确的版本 2. 实现乐观锁防止并发修改 3. 添加规则编辑锁定 4. 完善审计日志 |
| 规则复杂度失控 | HIGH | 1. 规则审查，识别重复和冗余规则 2. 实现规则重组，建立规则组 3. 提供规则复用机制 4. 建立规则评审机制 |
