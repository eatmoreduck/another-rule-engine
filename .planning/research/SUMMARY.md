# 项目研究总结

**项目:** 低代码风控规则引擎
**领域:** 风控规则引擎（Risk Control Rule Engine）
**研究日期:** 2025-03-26
**置信度:** HIGH

## 执行摘要

本项目是一个面向电商反欺诈场景的低代码规则引擎系统，其核心目标是让非技术人员能够通过可视化界面配置和管理风控规则，同时保证毫秒级的决策响应时间（<50ms）。研究表明，这类系统最关键的挑战在于性能优化、安全隔离和版本管理三个方面。

基于行业最佳实践（美团Zeus、有赞风控系统），推荐采用**Groovy DSL + Spring Boot 3.3 + Java 21**的技术栈，利用虚拟线程、脚本缓存和特征预加载实现高性能规则执行。架构上应采用**五层分层架构**（前端层、接入层、核心引擎层、执行层、存储层），核心组件包括规则解析器、流程编排器、版本管理器和灰度控制器。必须从第一天起就建立脚本缓存机制和沙箱隔离，否则后期修复成本极高。

风险方面最关键的是：**Groovy动态类加载导致的内存泄漏**、**缺乏沙箱隔离导致的代码注入攻击**、**规则执行超时导致的服务雪崩**，以及**灰度验证不充分导致的生产事故**。这些问题的修复成本远高于预防成本，必须在Phase 1就实现相应的防护机制。

## 关键发现

### 推荐技术栈

**核心技术栈（来自STACK.md）：**
- **Java 21 LTS** — 支持虚拟线程（Project Loom），显著提升高并发场景性能
- **Spring Boot 3.3.x** — 原生支持虚拟线程，启动时间减少40%，内存降低25%
- **Groovy 4.x/5.x** — 动态语言特性适合规则DSL，与Java生态无缝集成
- **PostgreSQL 16/17** — 规则元数据存储，支持JSONB存储规则配置
- **Caffeine + Redis** — 两级缓存策略，兼顾性能和分布式需求
- **React 19 + TypeScript** — 可视化规则编辑器，支持并发渲染和自动批处理

**版本兼容性关键点：**
- Java 21与Spring Boot 3.2+、Groovy 4.x+完全兼容
- Spring Boot 3.3.x支持Java 17+，但Java 21性能最佳
- PostgreSQL 16/17需要JDBC Driver 42.6.x+

### 预期功能

**必须具备（行业标准）：**
- **规则可视化配置** — 拖拽式流程编排，降低业务人员使用门槛
- **规则表单配置** — 简单规则快速配置入口
- **同步API执行** — 支付链路需要实时决策，目标<50ms响应时间
- **规则版本管理** — 支持版本创建、修改、删除、回滚
- **规则一键回滚** — 新规则出现问题需快速恢复，避免业务损失
- **规则灰度发布** — 按百分比流量切换，降低新规则上线风险
- **规则测试验证** — 支持模拟数据测试、实时调试
- **特征获取多策略** — 优先使用入参特征，降级到外部特征平台

**应该具备（竞争优势）：**
- **同步/异步混合执行模式** — 一套引擎两种模式，覆盖更广泛业务场景
- **决策表支持** — 多维度条件组合，适合评分卡、费率计算场景
- **规则效果分析** — 统计规则命中率、误判率、拦截率等指标
- **规则冲突检测** — 自动检测规则之间的逻辑冲突
- **规则依赖分析** — 分析规则之间依赖关系，避免冲突和死循环

**延后到v2+：**
- AI辅助规则生成（技术不成熟）
- 机器学习模型集成（增加复杂度）
- 实时流处理（v1仅支持同步API+异步事件）
- 移动端管理界面（v1仅Web端）

### 架构方法

**五层分层架构（来自ARCHITECTURE.md）：**

1. **前端层** — React 19 + TypeScript + Ant Design + React Flow，提供可视化流程编排和表单配置
2. **接入层** — 风控网关，负责鉴权、参数校验、幂等性、任务派发、记录落地、预警
3. **核心引擎层** — 规则解析器、流程编排器、版本管理器、灰度控制器
4. **执行层** — 同步执行器（GroovyShell）和异步执行器（事件驱动）
5. **存储层** — PostgreSQL（规则元数据）、Redis（运行时缓存）、MongoDB（审核记录）

**关键设计模式：**
- **责任链模式** — 规则执行链路处理，支持"命中即终止"逻辑
- **模板方法模式** — 定义规则执行的标准流程骨架（预处理→执行→后处理）
- **策略模式** — 不同规则类型的执行策略（统计规则、条件规则、自定义规则）
- **观察者模式** — 规则执行结果通知（记录日志、发送告警、更新统计）
- **工厂模式** — 规则实例创建

### 关键陷阱

**1. Groovy动态类加载导致内存泄漏**
- **问题：** 每次执行Groovy脚本时动态创建类，GC无法释放，最终耗尽Metaspace空间
- **预防：** 实现脚本缓存机制，避免重复编译相同脚本；使用单例ClassLoader；定期清理类加载器缓存
- **警告信号：** Metaspace使用率持续增长、频繁Full GC、大量Groovy动态类

**2. 缺乏沙箱隔离导致代码注入攻击**
- **问题：** 攻击者利用Groovy动态特性绕过沙箱限制，执行任意代码
- **预防：** 使用CompilerConfiguration自定义安全管理器；禁用危险类（System、Runtime、ProcessBuilder）；实现SecurityManager白名单机制
- **警告信号：** 规则包含getClass()、forName()等反射调用；尝试访问文件系统、网络、进程

**3. 规则执行超时导致服务雪崩**
- **问题：** 单个规则的恶意或错误配置（如死循环）导致执行超时，耗尽线程池
- **预防：** 为每个规则执行设置超时时间（50-100ms）；使用Resilience4j实现熔断机制；实现规则复杂度评估
- **警告信号：** P99耗时>50ms；线程池队列持续积压；某个规则执行时间>1s

**4. 特征获取性能瓶颈**
- **问题：** 规则执行过程中需要大量外部RPC调用或数据库查询，无法满足50ms延迟要求
- **预防：** 优先使用入参携带的特征；实现特征缓存机制（本地+分布式）；支持特征批量获取
- **警告信号：** 特征获取耗时占规则执行总时间50%以上；单个特征获取耗时>10ms

**5. 灰度验证不充分导致生产事故**
- **问题：** 新规则上线前未充分验证，直接全量发布，导致误伤正常用户
- **预防：** 实现灰度发布机制；灰度流量从小到大逐步增加（1%→5%→10%→50%→100%）；监控关键指标；实现一键回滚
- **警告信号：** 新规则上线后客诉激增；交易拦截率异常变化；灰度规则命中率过低

## 路线图启示

基于综合研究，建议采用以下阶段结构：

### 阶段1：核心规则引擎（基础设施）

**理由：** 必须首先建立稳定、安全、高性能的规则执行基础设施。根据PITFALLS研究，内存泄漏、沙箱绕过、执行超时等关键问题必须在第一天就解决，否则后期修复成本极高。

**交付物：**
- Groovy脚本编译和执行引擎
- 脚本缓存机制（防止内存泄漏）
- 沙箱安全隔离（防止代码注入）
- 超时熔断保护（防止服务雪崩）
- 特征获取多策略（入参优先+外部降级）
- 基础监控（性能指标、异常率）

**涵盖功能：**
- 同步API执行（<50ms目标）
- 规则持久化（PostgreSQL + JPA）
- 规则DSL存储（Groovy）
- 规则测试验证（模拟数据测试）

**避免陷阱：**
- Groovy内存泄漏（实现脚本缓存）
- 沙箱绕过漏洞（实现安全隔离）
- 规则执行超时（实现熔断机制）
- 特征获取性能瓶颈（实现多级缓存）

**使用技术栈：**
- Java 21 + Spring Boot 3.3.x
- Groovy 4.x/5.x
- PostgreSQL 16/17 + Spring Data JPA
- Caffeine本地缓存
- Spring Boot Actuator + Prometheus监控

### 阶段2：规则管理与版本控制

**理由：** 阶段1建立稳定的执行引擎后，阶段2重点解决规则的可管理性。版本管理、灰度发布、一键回滚是风控系统的核心安全机制，必须尽早实现。

**交付物：**
- 规则版本管理系统（创建、修改、删除、历史追溯）
- 规则一键回滚（秒级切换到历史版本）
- 灰度发布机制（按比例流量切换、自动回滚）
- 规则可视化配置UI（拖拽式流程编排）
- 规则表单配置UI（简单规则快速配置）
- 审计日志（操作记录、原因追溯）
- 规则导入导出（批量操作）

**涵盖功能：**
- 规则版本管理（来自FEATURES.md）
- 规则一键回滚
- 规则灰度发布
- 规则可视化配置
- 规则表单配置
- 审计日志

**避免陷阱：**
- 规则版本冲突（实现乐观锁）
- 灰度验证不充分（实现完整灰度流程）
- 规则复杂度失控（实现规则组管理）

**使用技术栈：**
- React 19 + TypeScript
- Ant Design + React Flow
- Spring Data JPA（版本管理）
- Redis（灰度流量计数）

### 阶段排序理由

1. **依赖关系：** 规则版本管理→规则灰度发布（灰度需要多版本共存）；规则版本管理→规则一键回滚（回滚需要历史版本）
2. **风险优先级：** 内存泄漏、沙箱绕过、执行超时是致命问题，必须在Phase 1解决
3. **价值交付：** Phase 1交付可用的规则执行能力，Phase 2交付完整的规则管理能力，Phase 3增强分析和优化能力
4. **技术复杂度：** 同步/异步混合模式需要稳定的基础设施，放在Phase 5避免过早复杂化

### 需要解决的缺口

**在实现过程中需要验证的领域：**
- **Groovy 5.x生产环境稳定性：** 研究显示Groovy 5.x也是稳定选择，但生产环境验证较少，需要在实际项目中验证

- **50ms延迟目标的可达成性：** 行业标准是<100ms，优秀系统优化到50-75ms，50ms目标需要精心优化特征获取和规则执行
